\chapter{Introduction}\label{chap:intro}

Software is written with intent, an intent to make a computer do something. Software
engineering, in essence, is the process of taking such intent, refining, reshaping,
reformulating it into a form that the computer understands. Software testing is the
bridge between the intent and the produced program; it is the process of tinkering,
measuring, understanding what the program does, separating it from how the program is
implemented.

Different methods of software testing force the programmer to express the intent in
different forms. In the simplest scenario, the programmers \emph{tinker} with the
program at hand. They interact with the produced artifact, pressing buttons in a user
interface, exploring different options in a command-line interface, testing a variety
of inputs for a function interacting through REPL... Going forward, these interactions
are typically \emph{codified}, written as concrete steps of testing instructions to be
run alongside the program development, run as part of an automated test suite of
hundreds or thousands of such interactions. The simplest form of codification is
\emph{example-based testing}, in which the specification relates concrete user written
input-output pairs together. A sorting function is tested through several assertions
such as \code{sort([]) == []}, \code{sort([1, 0]) == [0, 1]}, \code{sort(3,2,-1) ==
[-1, 2, 3]}...

Another form of specification, the one that will be of focus to this whole thesis, is
\emph{properties}. A property is a universally quantified predicate function. A fact
about the program under test that must hold for any possible input. The universality of
properties allows for relieving the user of the responsibility of supplying inputs.
Once a property is written as specification of a particular program behavior, there are
different strategies for producing inputs, systematic exhaustive search, random
sampling, perhaps even user-supplied inputs. The combination of these methods are
typically gathered under the umbrella of \emph{Property-Based Testing}.

Property-Based Testing (PBT) was pioneered by Koen Claessen and John Hughes in their
ICFP 2000 paper ``QuickCheck: a lightweight tool for random testing of Haskell
programs''~\cite{ClaessenH00}. The initial steps of random software testing is
typically attributed to the CS736 class of University of Wisconsin Madison Computer
Science Department in which Bart Miller coins the term ``fuzz'' testing, fuzzing UNIX
utilities with random inputs to produce breakages, triggering faults, finding bugs in
programs\cn. While the fuzz testing literature evolved with a focus on finding more and
more faults via smarter, guided, targeted input generation techniques\cn,
Property-Based Testing has chosen the specifications as its main focus. The original
paper introduces a language, as well as an implementation as a library, for expressing
properties, and writing random data generators; the design of which has been replicated
in virtually all programming languages in the last 26 years.
Table~\ref{tab:pbt-frameworks} gives a glimps of the reach in which QuickCheck has made
into the random testing world, the table presents the community-maintained
\code{jmid/pbt-frameworks} survey~\cite{jmid-pbt-frameworks}.

The ubiquity of the approach stems from its simplicity. The user starts with a simple
universal predicate denoting the correctness of their program:

\begin{code}[language=Haskell,emph={sort_correct,is_sorted,sort,is_permutation},emph={[2]l}]
sort_correct :: Ord a => [a] -> Bool
sort_correct l = is_sorted (sort l) && is_permutation l (sort l)
\end{code}

The \code{sort_correct} predicate denotes the correctness of a sorting function, the
elements of the list must not change, and the output must obey an ordering relation.
The user then writes a random data generator for the input type, as \code{sort_correct}
is defined over arbitrary lists of orderable elements, we definitely need a \code{List}
generator, and we can pick any orderable type for our purposes, which I picked
\code{Int}s in this case. QuickCheck provides a library of combinators for writing
generators such as \code{choose} and \code{listOf}, and the \code{Arbitrary} typeclass
for denoting generators of certain types.

\begin{code}[language=Haskell,emph={arbitrary,choose,listOf},emph={[2]a}]
instance Arbitrary Int where
    arbitrary = choose (0, 100)

-- listOf is a combinator that takes a generator for a
-- type and produces a generator for lists of that type.
instance Arbitrary a => Arbitrary [a] where
    arbitrary = listOf arbitrary
\end{code}

Once the universally quantified predicate and the corresponding random data generators
are in place, we can use QuickCheck's \code{forall} combinator for combining them into
properties to be tested.

\begin{code}[language=Haskell,emph={prop_sort_correct,forAll,arbitrary,sort_correct,quickCheck},emph={[2]xs}]
prop_sort_correct :: Property
prop_sort_correct = forAll arbitrary $ \xs -> sort_correct xs

quickCheck prop_sort_correct
{-
Stdout:
+++ OK, passed 100 tests.
-}
\end{code}

What happens in the background after the \code{quickCheck} call? Well, the simplest
form of a Property-Based Testing runner is as follows: A \generator{} produces some
values a \checker{} tests and produces one of three possible results, either a
\passed{}, indicating that the predicate executed with the input did not reveal any
bugs, \discarded{}, indicating that the input is not a valid input for the property,
hence was promptly discarded, or \failed{}, indicating that the input constitutes a
bug, a violation of the property. After reaching a failing input, the second stage of
the property runner is initiated, the bug is minimized through shrinking with the
application of delta-debugging\cn. A \shrinker{} produces multiple candidate shrunk
values, sometimes by shrinking an internal value such as keys in a tree, or removing
parts of the structure such as removing part of a computation. If a candidate
reproduces the failure, the shrinking is recursively applied to the smaller candidate
until there are no more failing candidates, at which point the minimal value is
\printer{printed}. An abstract form of this process is depicted in
Figure~\ref{fig:quickcheck-property-runner-intro}:

The implementations presented in the PBT frameworks table
(Table~\ref{tab:pbt-frameworks}) all implement different versions of this same core
algorithm with changes to the internal parameters to the generators and shrinkers, the
APIs for expressing the properties, the generators, the shrinkers as well as the
underlying source of randomness. While Hypothesis~\cite{Hypothesis}, Zest~\cite{Zest},
Crowbar~\cite{Crowbar} parse randomness~\cite{goldsteinParsingRandomness2022} into
structures in their respective programming language, many frameworks use QuickCheck
style RNG splitting\cn{} that starts from an initial random seed and splits it at every
decision point.

While many of these decisions are defaults that can be overridden as users require
custom configurations for their domains, many of them \emph{leak} outside the
abstraction and may require significant reengineering of the library internals to
accommodate alternative behaviors. For instance, the decision to use randomness parsing
vs. RNG splitting affects one of the columns in the framework table
(Table~\ref{tab:pbt-frameworks}), \emph{integrated shrinking}. In integrated
shrinking~\cite{HypothesisShrinking}, in-comparison to the classic structural shrinking
approach we mentioned, the source of randomness that is parsed to the generated
structure is shrunk instead of the structure itself. As such, a library using RNG
splitting cannot accommodate integrated shrinking without reengineering of its
internals.

Chapter~\ref{chap:ppbt} on Programmable Property-Based Testing~\cn{} focuses on this
exact problem and its many variants that cause fragmentation of the PBT landscape.
Table~\ref{tab:pbt-frameworks} presents 50 frameworks in 31 languages, with up to 5
libraries in OCaml. We trace a precise design decision, the design of the property
language in QuickCheck, as the predominant source of inflexibility of a PBT framework,
and the primary cause of ecosystem fragmentation as the initial design, a shallow
embedded domain-specific language, forces the implementation of the property runner to
reside in the ``library-space'' as opposed to ``user-space''. We present an alternative
property-language, a mixed-embedding, that enables flexible user-space property
runners, allowing us to combine many features and advances dispersed throughout the
ecosystem under a single library that users can keep on extending.

Programmable PBT addresses a huge challenge, the ability to consolidate algorithmic
advances easily within a single framework instead of significant reengineering of
internals, forking the original library into a new path, or even further so, writing a
new library from scratch. It lowers the cost of experimentation, speeding up scientific
experimentations, enabling diffusion of newer advances throughout the ecosystem.
However, fragmentation isn't just in-language frameworks competing with each other,
it's more so an information dissemination problem. How would frameworks include a novel
approach without concrete empirical understanding of its effects within their context?
A brief study of \todo{K} PBT papers taken from the PBT bibliography\cn{} reveals a
core issue, quantitative evaluation of PBT has been dispersed through bespoke,
purpose-specific benchmarks implemented for the evaluation sections of those papers, to
be forgotten once they are accepted.

In Chapter~\ref{chap:etna}, I introduce ETNA~\cite{ETNA}, the first large-scale
evaluation platform for PBT. We designed ETNA to be easily extensible in terms of
adding new testing strategies for evaluation, new workloads as novel benchmarks, and
new languages with entirely separate ecosystems from the existing ones. The purpose of
ETNA is to consolidate knowledge through a common interface on quantitative evaluation,
same benchmarks through various languages to compare across-language performance
trade-offs, sampling based evaluation modes for abstracting away language-specificness
of the frameworks and further focus on the quality of the generated tests, reproducible
experiments for easily sharing, verifying and replicating existing results. Similar to
Programmable PBT, ETNA is a step towards accelarating scientific development of PBT.

Much of the contemporary discussion of features, generalization, performance,
interface, and usage of Property-Based Testing is based on general purpose frameworks
we have outlined in Table~\ref{tab:pbt-frameworks}. However, the world is full of
oddities, special-purpose tools, requirements that push outside the boxes we draw.
General purpose tools grow as they redraw the lines of their approach to accommodate
for such extensions. In a thesis focusing on consolidation, it is only natural that I
take the time to focus on what we can not, at least for the moment, consolidate. In
Chapter~\ref{chap:dirt}, I present and discuss a domain-specific property-based testing
paradigm for databases, ``DIRT: Database-Integrated Random Testing''~\cn. Modern
databases have enjoyed the results of fuzzing, at times with property-based testing
flavored approaches, in strenghtening their reliability and correctness. However, a
major challenge that is yet to be solved is the ability to evolve a property-based
testing suite alongside the database, which we attempted to solve in our \todo{DBTest}
paper. The chapter discusses various other domain-specific tools and techniques as well
as their potential inclusion to general purpose PBT frameworks.
